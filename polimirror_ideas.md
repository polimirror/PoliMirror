# 🪞 PoliMirror アイデアログ

> 深淵をのぞく時、深淵もまたこちらをのぞいているのだ

**ミッション:** 政治家の言葉を、すべて記録する。それだけで、政治は変わる。

このファイルは思いついたアイデアをすべて溜めておく「生きたメモ」です。
実装済み・未実装関係なくどんどん追記します。

## 📋 情報記録原則（絶対に守る・最重要）

**PoliMirrorは「事実の鏡」。噂・疑惑・都市伝説は一切記録しない。**

### 記録するもの
```
✅ 本人が発言・公表した事実（出典付き）
✅ 公式文書・議事録・官報に記載された事実
✅ 裁判・行政処分など公的に確定した事実
✅ 報道機関が署名入りで報じた事実
✅ 疑惑・告発があり「無実が確定した」事実（※後述）
```

### 絶対に記録しないもの
```
❌ 噂・まとめサイト・匿名情報
❌ 疑惑段階のものを事実として書く
❌ 「〜の可能性がある」「〜と言われている」
❌ SNSの拡散情報（出典が辿れないもの）
❌ 都市伝説・陰謀論の類
❌ 確定していない捜査情報・リーク情報
```

### 「無実が確定した疑惑」こそ記録する
> これがPoliMirrorの最も価値ある機能の一つ

```
よくある構造:
疑惑が拡散 → 政治家のイメージが傷つく
　↓
裁判・調査の結果「無実」が確定
　↓
でも「無実」の方はほとんど拡散されない
　↓
ネット上には「疑惑」だけが残り続ける

PoliMirrorがやること:
「疑惑があった」事実 ＋ 「無実が確定した」事実
を両方・同等の重みで記録する
```

**具体例（大阪・上海電力問題）**
```
疑惑: 「上海電力を導入した＝親中」と拡散された
調査: 大阪市・府の入札記録・契約書を精査
結果: 正規の入札手続きに基づく契約であることが確認
　　　「親中」を示す証拠は存在しなかった

PoliMirrorの記録:
・疑惑が拡散した事実（日時・媒体）
・調査・検証の経緯
・無実が確定した事実（出典：調査報告書・裁判記録）
・「拡散された疑惑は事実ではなかった」と明記
```

**なぜ「無実の記録」が重要か**
```
政治家を守るためでも攻撃するためでもなく
「真実がどこにあるか」を示すため

デマで傷ついた政治家の名誉回復にも使える
→ 信頼性の高さがPoliMirrorの最大の資産になる

デマを流した側の「訂正しない」という事実も
別途記録できる（これ自体が重要なデータ）
```

### 信頼度スコアの設計
```
★★★★★ 国会議事録・官報・裁判確定記録
★★★★☆ 主要報道機関の署名記事
★★★☆☆ 地方紙・業界紙の署名記事
★★☆☆☆ 週刊誌・ネットメディア（実名記者あり）
★☆☆☆☆ 使用しない（匿名・まとめ・SNS拡散）
```

### 未解決疑惑の記録ポリシー（案B＋C＋D）

> 裁判で決着がついていない・本人SNSで釈明のみの場合

**記録する条件**
```
・主要報道機関（★★★以上）が実名・署名で報じた
・または本人が公式に言及・釈明した
  （本人が言及した時点でその事実自体は記録対象になる）
```

**記録する内容（3点セット・必ず揃える）**
```
① 「〇〇の報道があった」
   → 出典明記・信頼度★表示

② 「本人の釈明・対応」
   → 出典明記（SNS投稿・会見・議会答弁等）
   → 釈明がない場合は「本人からの公式コメントなし」と明記

③ 「現時点で公式な決着（裁判・行政調査等）はついていない」
   → これを必ず明記する
   → 決着がついた時点で自動的にステータスを更新する
```

**記録しない条件**
```
・SNS拡散・まとめサイト・匿名情報が出所
・報道があっても★★以下の単独報道のみ（原則記録しない）
・本人が一切言及しておらず報道も★★以下
```

**ステータス管理**
```
未解決   → 「報道あり・決着未了」と表示
有罪確定 → 「裁判確定（有罪）」に更新
無罪確定 → 「裁判確定（無罪）」に更新・疑惑報道とセットで表示
取り下げ → 「報道取り下げ・訂正あり」に更新
```

**最重要原則**
```
「決着がついていない案件は
 決着がついていないと明記する」

有罪でも無罪でもなく「現在進行形」であること自体が情報。
解釈はユーザーに完全に委ねる。
PoliMirrorは判断しない。事実を並べるだけ。
```

## 📌 プロダクト基本設計

- **対象:** 日本の政治家35,000人（国会議員・知事・市長・都道府県議・市区町村議・落選者・候補者・元議員）
- **コンセプト:** 監視でも批判でもなく「透明化」。良い政治家が正当に評価され、そうでない政治家が隠せなくなる
- **設計原則:** 全出典付き・事実のみ・プラスもマイナスも等しく記録・解釈はユーザーに委ねる
- **1議員 = 1MDファイル** 自動生成・自動更新
- **展開:** Obsidian（思考）/ GitHub（履歴）/ GitHub Pages + SaaS（公開・収益）

### 宗教団体との関わり系（最もセンシティブ・最も重要）
> 統一教会問題以降、有権者の関心が急騰。出典の厳格管理が命。

**記録する条件（厳格に守る）**
```
✅ 本人が公式に認めた関係
✅ 議会・国会での質疑応答で明らかになった事実
✅ 政治資金収支報告書に記載された寄付・献金
✅ 主要報道機関が署名記事で報じた事実（★★★以上）
✅ 宗教団体主催イベントへの出席（公式記録・写真）

❌ 「〇〇教と関係がある」という噂・まとめ
❌ 宗教団体が「支援している」と一方的に発表した情報のみ
❌ 家族・親族の信仰（本人の行動でない限り）
❌ 未確認の献金情報
```

**記録するデータ**
```
接点の種類:
- 祝電・メッセージ送付（イベント名・日時・出典）
- イベント登壇・来賓出席（公式記録）
- 政治資金としての受取（収支報告書）
- 選挙応援・組織票の受け入れ（公式発表・報道）
- 政策協定・要望書への署名

対象宗教団体:
- 特定の団体に絞らず全宗教・全団体を同じ基準で記録
- 神社本庁・創価学会・統一教会・その他すべて平等に
- 「この宗教は問題」という判断はしない

議員の対応記録:
- 関係を認めたか否か（発言の記録）
- 関係発覚後に関係を断ったか（日時・方法）
- 返金・返却したか（金額・時期）
- 謝罪・説明会見の有無
```

**設計上の最重要原則**
```
宗教との関係を記録することは
宗教を批判することではない

特定宗教だけを標的にしない
与野党・全議員を同じ基準で記録する
「関係があった」という事実だけを記録
善悪の判断はユーザーに完全に委ねる

→ これを守らないと
　 PoliMirrorの中立性が崩壊する
　 特定の政治勢力の道具になる
```

### 知事交代による変化の記録系（地方政治の継続性評価）
> 「誰が知事になったら何が変わったか」を数値で記録する。

**記録するデータ**

```
政策の継続・変更:
- 前知事の主要施策を継続したか・廃止したか
- 廃止した場合: 理由（公式）・コスト（埋没費用）
- 新設した場合: 予算規模・目標値

数値で変化が見えるもの:
- 財政（借金残高・基金残高・税収）
- 人口（増減・転入転出）
- 企業誘致件数・雇用数
- 待機児童数・保育所数
- 公共交通の路線数・利便性
- 病院・診療所の数
- 独自の給付・補助金の新設・廃止

良くなった事例の記録:
└─ 財政再建に成功した知事
└─ 企業誘致で雇用が増えた県
└─ 独自政策で人口減少を緩やかにした市
└─ 行政デジタル化で住民サービスが向上

悪くなった事例の記録:
└─ 大型事業の頓挫で財政悪化
└─ 産業政策の失敗で企業流出
└─ 首長交代で継続事業が中断→損失発生
└─ 職員士気の低下・不祥事の増加
```

**「知事交代ビフォーアフター」フォーマット**
```
【〇〇県・知事交代 20XX年】

前知事（在任〇年）
  財政: 借金〇億円 → 〇億円（+〇億円）
  人口: 〇万人 → 〇万人（-〇人/年）
  主要施策: 〇〇計画（達成率〇%）

現知事（在任〇年）
  財政: 〇億円 → 〇億円（+/-〇億円）
  人口: 〇万人 → 〇万人（-〇人/年）
  主要施策: 前知事施策を廃止/継続/変更
  新施策: 〇〇（予算〇億円・達成率〇%）

変化のサマリー（事実のみ）:
  良くなった: 〇〇（出典・数値）
  悪くなった: 〇〇（出典・数値）
  変化なし:   〇〇
```

### 自治体予算・計画の追跡系（地方政治の核心）
> 首長・地方議員の「実績」を数値で測る唯一の方法。言葉ではなく結果で評価する。

**予算の収支追跡**
```
当初予算 vs 決算額（毎年度）
- 余剰額・不足額（絶対値・率）
- 余った場合: 基金積立か？翌年度繰越か？
- 足りなかった場合: 補正予算の回数・規模
- 借金（地方債）残高の推移
- 積立金（基金）残高の推移
- 財政健全化判断比率（国が定める危険水域との比較）

自治体間比較:
- 人口1人あたりの予算規模
- 人口1人あたりの借金残高
- 同規模自治体との偏差
```

**頓挫した計画の記録**
```
記録する条件:
- 議会で可決された事業が中断・廃止された
- 首長の公約として発表されたが実現しなかった
- 予算を執行したが目標未達で事業終了

記録する内容:
- 事業名・当初の目標・予算額
- 執行額（無駄になった金額）
- 頓挫した理由（公式発表）
- 責任者（当時の首長・担当部局）
- 議会での審議経緯

典型例:
└─ 大型公共施設の建設中止（設計費だけ消えた）
└─ 観光施設・道の駅の赤字運営→廃止
└─ IT化・DX事業の失敗
└─ 人口増加目標「2030年までに〇万人」→達成できず
```

**成功した計画の記録**
```
記録する条件:
- 数値目標が達成された
- 第三者評価・監査で成果が確認された
- 当初予算内で完了した

記録する内容:
- 事業名・目標・実績値
- 予算額・執行額・余剰額
- 完了年度・担当首長・議会
- 他自治体への波及（モデルケース化）

典型例:
└─ 待機児童ゼロ達成（いつ・何人削減・予算）
└─ 財政再建（借金〇億円削減・何年で）
└─ 移住促進（目標〇人→実績〇人）
└─ 独自の産業振興策の成功
```

**首長・議員との紐付け**
```
予算を承認した → 議会・議員
予算を執行した → 首長・副知事・部局長
結果の責任者  → 在任中の首長に帰属

「この首長の在任中に
 〇億円の事業が頓挫した」
「この議員は賛成票を投じた」
が数値付きで永久に記録される
```

```
「子育て支援を充実させます」
「財政を健全化します」

でも結果は数値で出る

・頓挫事業の累計損失額

「言葉ではなく数字で評価する」
```

### 疑惑ステータス管理システム
> 疑惑の一生を記録する。「選挙後に答える」が本当に答えられたか追跡する。

**疑惑のライフサイクル**
```
発生 → 追及 → 回答/回避 → 結果
このフロー全体を記録する
```

**ステータス一覧**
```
🔴 疑惑浮上中     → 報道・国会で指摘・本人コメントなし
🟡 回答待ち       → 「調査します」「確認します」期限未定
🟠 期限付き回答待ち → 「○月○日までに報告」カウントダウン表示
⚫ 期限切れ・未回答 → 約束した期限を過ぎた・最重要
🔵 選挙後に回答予定 → 「選挙が終わったら」と発言・追跡対象
🟢 回答済み       → 内容・日時・回答の質を記録
⚪ 有耶無耶・立ち消え → 誰も追及しなくなった・でも記録し続ける
```

**回答の質スコア（Claude API判定）**
```
A: 完全回答  → 具体的な事実・数字・経緯を説明
B: 部分回答  → 一部だけ答えた
C: 論点ずらし → 別の話にすり替えた
D: 形式的回答 → 「真摯に対応します」で終わった
F: 事実上の無回答 → 「記憶にない」「確認中」のみ
```

**経過日数カウンター**
```
⚫ A社献金問題: 未回答 【847日経過】
🟠 B社接待問題: 回答期限まで【残り3日】
⚫ C氏口利き疑惑: 未回答 【1,203日経過】

→ 日数が増えるほど「忘れた頃に思い出させる」効果
```

**「選挙後に答える」の追跡**
```
選挙後に実際に答えた確率を統計化
→「選挙後に答えると言った議員が
  実際に答えた確率: 12%」

→ この数字が出た瞬間に
  「選挙後に答える」という言葉の
  本当の意味が証明される
```

**お金の問題の類型**
```
① 収支報告書の記載漏れ
② パーティー券の裏金化
③ 寄附の不記載
④ 政治資金の私的流用疑惑
⑤ 献金元企業との利益相関

「山田議員の政治資金疑惑:
 総額2,300万円・現在⚫未回答（847日経過）」
```

**情報収集ソース**
```
取れるもの:
✅ 国会議事録（委員会・本会議での追及と回答）
✅ 記者会見（Whisperで文字起こし）
✅ 議員個人のSNS・公式サイト

取れないもの:
❌ 週刊誌・テレビ → 間接記録としてのみ保存
```

### 公約・マニフェスト達成率管理
> 「言ったことをやったか」を選挙ごとに追跡する。曖昧な公約は曖昧と明記する。

**公約データの収集方法**
```
選挙公報（総務省・各選管が公開）
→ 全候補者の公約が全部掲載
→ PDFをOCRでテキスト化

政党マニフェスト（各党公式サイト）
→ 過去分はWebArchiveから取得

議員個人の選挙サイト
→ 選挙中だけ公開されることが多い
→ 選挙公示日にクローラーを走らせて必ず取得・保存
```

**選挙ごとの自動アクション**
```
選挙公示日:
→ 全候補者の公約を自動取得・保存・SS撮影

投開票日:
→ 当落結果を記録
→ 当選者の公約を「追跡対象」に設定

当選後毎年:
→ 公約達成状況を自動チェック
→ 国会議事録・法案成立記録と照合

次の選挙前:
→ 達成率レポートを自動生成・公開
```

**判定の3段階**
```
✅ 明確に達成
   数字・法案・予算で証明できる

⚠️ 判定困難 = 曖昧な公約
   基準が不明確・何をもって達成とするか不明
   → 判定困難の理由も明記する
   → 「曖昧な公約を書いた」という事実を記録

❌ 明確に未達成
   反対の政策を実施した等
```

**「判定困難＝曖昧な公約」の可視化**
```
今まで誰も指摘しなかった視点

「景気を良くします」→ 判定困難 = 曖昧な公約

これを明記するだけで:
・有権者が公約の質を評価できる
・曖昧公約が恥ずかしいものになる
・政治家が具体的な公約を書くようになる
・曖昧にすること自体が責任逃れの手段だったと証明される
```

**公約の質スコア（Opus APIで判定）**
```
5点: 数字・期限・手段が全部ある
     例「消費税を2025年までに5%に下げる」

4点: 数字か期限のどちらかがある
     例「最低賃金を1,500円にする」

3点: 方向性は明確だが数字がない
     例「最低賃金を引き上げる」

2点: 方向性が曖昧
     例「経済政策を見直す」

1点: 何も言っていないのと同じ
     例「景気を良くします」
        「国民のために働きます」
        「政治を変えます」
```

**曖昧公約ランキング**
```
「最も曖昧な公約を掲げた議員」
「最も具体的な公約を掲げた議員」

→ 選挙前に公開すると有権者の投票基準が変わる
→ 長期的に政治の質が上がっていく
```

**宣言と結果のマトリクス**
```
           具体的公約    曖昧公約
達成した  │    ◎      │    ?    │
未達成    │    ✕      │  測定不能│
判定困難  │   まれ    │   多い  │

→「曖昧公約は達成率を測定できない
  仕組みになっている」が一目でわかる
```

### 判定モデルの原則（透明性の核心）
> 判定の質 = PoliMirrorの信頼性。常に最高のモデルを使い、それを明示する。

**「判定には常に最高のモデルを使う」原則**
```
安いモデルで判定を間違えると:
・議員から「判定がおかしい」と攻撃される
・信頼性が崩れる
・修正コストが判定コストより高くなる

→ 判定だけは妥協しない
→ その時点での最高モデルを使う
```

**モデルの使い分け設計**
```
【最高モデル（Opus系）】
・公約の質スコア判定
・疑惑への回答の質判定
・矛盾・ダブスタ検出
・常套句の文脈判定
・発言の本気度判定
→ 結果がPoliMirrorの信頼性に直結するもの

【中間モデル（Sonnet系）】
・データの構造化
・MDファイルの自動生成
・要約・サマリー作成
・一次スクリーニング
→ 大量処理・精度より速度

【軽量モデル（Haiku系）】
・単純な分類・タグ付け
・重複チェック・フォーマット変換
→ 単純作業・コスト最小化
```

**コスト管理との両立**
```
Sonnetで一次判定
→ 判定困難・スコア3前後のものだけOpusで二次判定
→ コストを10分の1に抑えながら精度はほぼOpus同等

年間Opusコスト: 数十万円
でもそれがPoliMirrorの「信頼性」という資産になる
```

**判定モデルの明示（最重要）**
```
判定結果の下に必ず表示:

「この判定はClaude Opus 4.6
 （2025年○月時点）により行いました」
「判定ロジック: GitHub参照」

→ 判定の根拠が明確になる
→ AIが恣意的に判定したと言えない
→ 人間の主観が入っていないことの証明
→ モデルが変わったら再判定できる
```

**バージョン管理との組み合わせ**
```
GitHub上で全部記録:

commit: 2025-03-15
「山田太郎の公約をOpus 4.6で再判定
 スコア: 2→3点に変更
 理由: モデル精度向上により文脈の読み取りが改善」

→ 判定が変わった理由も透明化
→「気まぐれで変えた」と言われない
```

**将来への対応**
```
2024年: Opus 3が最高
2025年: Opus 4.6が最高
2026年: さらに上のモデルが出る

→ 判定モデルを更新するたびに過去の判定を再評価
→「2024年の判定を2026年の最高モデルで再評価した結果」
→ 判定精度の向上が透明化される
→ これもGitHubで記録
```

### 良い議員の可視化（PoliMirrorの魂）
> 「良いことはニュースにならない」を変える。感情ではなくデータで証明する場所。

**なぜ良い議員が埋もれるか**
```
メディア:「良いことをした」→ ニュースにならない
SNS:「いい議員だ」とポストする → 「信者」と言われる
→ 真面目にやっている議員が正当に評価される場所がない
```

**PoliMirrorが変えること**
```
「信者じゃない。データがそう言っている」

これが言える場所は世界中にPoliMirrorしかない
感情ではなく数字で証明できる
```

**記録できる「良いこと」**
```
発言系:
・公約を言い続けている一貫性
・反対意見にも真摯に答えている
・データを使って議論している
・専門的な質問を繰り返している

行動系:
・委員会出席率95%以上
・質問主意書を毎月出している
・現場訪問の記録（被災地・福祉施設等）
・地元報告会の開催記録

お金系:
・政治資金を自主的に詳細公開
・パーティーをやっていない
・議員宿舎を使っていない
・政務活動費を全額返還した

結果系:
・実現した公約の数
・通過させた法案
・変えた条例
・陳情への対応記録
```

**埋もれた良い議員の発掘**
```
地方議員に特に刺さる

国会議員と違ってメディアに出ない
でも地道にすごい仕事をしている人がいる

「誰も見ていないと思っていたが
 PoliMirrorが全部見ていた」

→ 本人のモチベーションになる
→ 有権者が気づける
→ 次の選挙で正当に評価される
```

**点数化の難しさはGitHubで解決**
```
点数の計算式・重み付けに
正解はない・議論が分かれる

だからGitHubで全公開する

計算式を公開することで:
・「なぜこの重み付けか」の議論ができる
・コミュニティが改善できる
・恣意的と言われない
・バージョン管理で変更履歴も残る

「スコアに異議があれば
 GitHubでPull Requestを送ってください」
→ 民主的なスコア設計
```

**これがPoliMirrorを中立にする核心**
```
批判だけのサイト:
→ 政治的に偏っていると見なされる
→ 信頼されない

良い記録と悪い記録が同じ重みで存在する:
→ 与野党どちらの議員にも価値がある
→ 誰も「偏っている」と言えない
→ 良い議員を記録することが
  悪い議員への最大の圧力になる
```

### 透明性スコア設計（ズル検出器）

### データ収集ポリシーの透明化
> 収集方法を全公開することが最強の防衛になる。やましいことは何もない。

**収集しているのは全部公開情報**
```
国会議事録         → 国が公開
裁判所判決文       → 裁判所が公開
政治資金収支報告書  → 総務省が公開
議員公式サイト     → 本人が公開
YouTube公式チャンネル → 本人が公開

「公開されている情報を集めているだけ」
やましいことは何もない
```

**収集ルール（明文化・公開）**
```
① robots.txtを遵守する
② リクエスト間隔を5〜10秒に設定
   サーバーに負荷をかけない
③ 深夜帯のみ収集（負荷分散）
④ キャッシュを活用して再取得を最小化
⑤ ログインが必要なページは収集しない
⑥ 公務に関する情報のみ収集
   私生活・家族情報は収集しない
```

**収集コードをGitHubで全公開**
```
これが最強の回答

「どうやって収集しているか？」
→「GitHubで全部公開しています」
→「誰でも確認できます」
→「隠していることは何もありません」

収集コードの公開 = 透明性の証明 = 攻撃できない
```

**法的根拠**
```
・著作権法第13条（国会議事録等は著作権なし）
・情報公開法（公的機関の情報は公開義務あり）
・公開情報のスクレイピングを禁じる法律は日本にない
・不正アクセス禁止法 → ログインページは収集しないので対象外
・個人情報保護法 → 公務情報のみ収集するので対象外
```

**「なぜ問題なのか」と逆に問える**
```
「公開情報を集めることがなぜ問題なのですか？」
→ 答えられる人はいない

「この情報を集めてほしくないのは誰ですか？」
→ 圧力をかけてきた側の意図が透けて見える
```

```
当サービスは公開情報のみを収集します。
収集方法はGitHubで全公開しています。
robots.txtを遵守し、サーバーに負荷をかけない

収集目的は公益・政治の透明性向上です。
```

### 党員数データ
> リアルタイムは不可能だが「常に最新・推計値あり」で誰もやっていないコンテンツになる。

**取れるものと取れないもの**
```
取れるもの:
✅ 政治資金収支報告書の党費収入（年1回・総務省公開）
✅ 各党の公式発表（プレスリリース・大会資料）
✅ 国政選挙の比例得票数（支持者数の代替指標）

取れないもの:
❌ リアルタイムの党員数
❌ 月次・週次の増減
→ 各政党がそもそも公開していない
```

**PoliMirrorならではの推計値**
```
党費収入 ÷ 党費単価 = 推計党員数

例: 自民党
党費収入: 政治資金収支報告書から取得
党費単価: 公開されている党則から取得
→ 推計党員数を算出

誰もやっていない
「推計党員数ランキング」が作れる
```

**設計方針**
```
「リアルタイム更新」ではなく
「最新データに自動更新」

総務省が収支報告書を公開したら即取得・即反映
→ 年1回でも「常に最新」は維持できる
→ 更新日時を明示することで信頼性を担保
```

**時系列で見えてくるもの**
```
党員数の増減 × 政治イベントの相関

スキャンダル後に党員が減ったか
選挙に勝った後に党員が増えたか
政権交代前後の変化

→ 政党の「リアルな勢い」が数字で見える
```

### 法的防衛設計＆削除要請の透明化
> 一次ソースのみという原則が最強の盾。削除要請自体を武器に変える。

**なぜ一次ソースが盾になるか**
```
名誉毀損の成立要件に「真実でない」がある
PoliMirrorが記録するのは:
・国会議事録（著作権法第13条で転載自由）
・官報・裁判確定記録
・議員本人の公式発言

→「真実でない」と言えない
→ 名誉毀損が成立しない
```

**想定される攻撃と防衛**
```
「削除しろ」
→「出典は国会議事録○年○月○日です。
  削除するには国会議事録を書き換えてください」

「名誉毀損だ」
→「全て一次ソースからの引用です。
  虚偽であれば国会・官報・裁判所に申し立てを」

「プライバシー侵害だ」
→ 政治家の公務情報は知る権利・公益性が優先
  最高裁判例で確立済み

「著作権侵害だ」
→ 国会議事録は著作権法第13条で保護対象外
  自由に引用・転載できる
```

**弱点と対策**
```
週刊誌・SNSからの引用 → 使わない・一次ソースのみ徹底
解釈・分析部分 → 「AIによる分析です」と明示・事実と分析を分離
データ入力ミス → 修正フローを公開・修正履歴も全公開
```

**問い合わせ対応の現実**
```
99%はAIが即答で終わる
削除要請・訴訟キーワードが来た時だけ管理者に転送

ほとんどAIの返答で完結する設計で十分
```

**削除要請の透明化＋X投稿（重要）**
```
削除要請が来たら:
① 要請内容・要請者・日時を記録
② サイト上に「削除要請ログ」として公開
③ 対応結果も公開
   「一次ソースのため掲載継続」
   「誤りが確認されたため修正」
④ 即座にXにポスト

「〇〇議員から削除要請がありました。
 出典は国会議事録のため掲載を継続します。
 #PoliMirror」

→ 要請した議員の方が世論から批判される
→ 攻撃すればするほどPoliMirrorの知名度が上がる
→ 削除要請がそのまま拡散コンテンツになる
```

**最強の防衛は透明性そのもの**
```
出典を全部公開
修正履歴を全部公開
方針・ポリシーを全部公開
削除要請ログを全部公開

→ 攻撃する側が「隠している」と言えない
→ 攻撃すること自体が
  「記録を隠したい」という自白になる
```

**実務的な準備**
```
① 免責事項ページ
「一次ソースのみ掲載・事実と分析を分離・誤り指摘に真摯に対応」

② 弁護士顧問契約
月1〜3万円で心理的安全性が全然違う
削除要請が来た時に即座に対応できる体制

③ 削除要請ログの公開ページ
要請者・日時・内容・対応結果を全て記録
```

**結論**
```
一次ソースのみ + 透明性の徹底 = 法的にほぼ無敵

攻撃すればするほどPoliMirrorが強くなる構造
訴訟になっても裁判自体がニュースになり
PoliMirrorの知名度が上がる
```

### 公開インフラ設計: Quartz + GitHub + Cloudflare Pages
> 無料・高速・絶対に落ちない。PoliMirrorの公開層として完璧にフィット。

**3つの役割**
```
Quartz（OSS）:
→ ObsidianのMDファイルをWebサイトに変換
→ [[リンク]]がそのままWebのリンクになる
→ グラフビューもブラウザ上で動く
→ ユーザーがノードをクリックして探索できる

GitHub（publicリポジトリ）:
→ MDファイルの保管・バージョン管理
→ 「いつ・何を・どう変えたか」が全記録される
→ 透明性の証明（PoliMirrorの思想と完全一致）
→ AIクローラーに全文公開できる

Cloudflare Pages:
→ GitHubと連携・自動デプロイ
→ 世界300箇所以上に分散配信
→ 独自ドメイン対応・無料
```

**絶対に落ちない理由**
```
静的ファイルをCloudflareが世界中に分散して持つ

アクセスが集中しても:
→ 自動で負荷分散
→ 「サーバーダウン」という概念がない
→ Wikipedia級のトラフィックが来ても無料・無制限
→ 選挙当日に100万人アクセスしても落ちない

これが動的サーバー（VPS・AWS等）との決定的な違い
```

**自動更新フロー**
```
DBからMD自動生成（Python）
        ↓
GitHubにpush
        ↓
Cloudflare Pagesが自動検知・ビルド
        ↓
Quartzが変換
        ↓
polimirror.jp に数分で反映

→ 議員データを更新するだけでサイトに即反映
→ デプロイ作業ゼロ
```

**コスト**
```
Quartz:           無料（OSS）
GitHub:           無料（publicリポジトリ）
Cloudflare Pages: 無料（商用利用・無制限トラフィック）
独自ドメイン:     年間約1,500円のみ
```

**段階的展開設計**
```
フェーズ1（今すぐ）:
全データをQuartz + GitHub + Cloudflare Pagesで無料公開
→ LLMのクローラーに拾わせる
→ 日本政治版Wikipediaのポジションを取る

フェーズ2（収益化後）:
有料機能のみNext.js SaaSを追加
→ アラート・API・ダウンロード販売
→ 無料公開層はそのまま維持

無料層が集客→有料層でマネタイズ
の好循環設計
```

**注意点**
```
Quartzは静的サイト
→ 検索・グラフ表示はJSで動く（問題なし）
→ ユーザーログイン・課金は別途SaaS層が必要
→ ただし最初は不要・公開だけで十分
```

### 消費税還付企業トップ100（政治×投資の核心データ）
> 消費税還付は「輸出企業への実質補助金」。受取額上位企業と政治の関係は直結する。

**消費税還付とは**
```
輸出企業は国内で消費税を払うが
輸出先では消費税を取れない

→ 国が差額を還付する仕組み
→ トヨタ・ソニー等の大手輸出企業が
　 毎年数千億円単位で受け取っている

これは公開情報（国税庁・財務省）だが
構造化されたデータベースは存在しない
```

**記録するデータ**
```
還付企業×金額系:
- 消費税還付額ランキング（年別）
- 業種別の還付総額
- 還付額の推移（消費税率変更前後の変化）
- 還付額上位100社リスト（出典: 国税庁公開情報）

政治との接点:
- 還付上位企業の政治献金額・献金先議員
- 還付上位企業の業界団体と族議員の関係
- 消費税率変更の審議時の発言×還付企業との関係
  →「消費税増税に賛成した議員の献金元が還付上位企業」
  　 という構造が可視化される

投資シグナルとしての活用:
- 消費税政策の変化→還付額への影響シミュレーション
- 還付上位企業×為替政策発言（円安=還付増）
- 「減税議論が活発化→還付上位企業株に注目」
```

**なぜ重要か**
```
消費税の逆進性議論では「低所得者への負担」が語られる
でも還付の話はほとんど報道されない

還付トップ100 × 献金先議員 × 消費税賛成票
を並べるだけで構造が見える

PoliMirrorの「事実を並べるだけで政治は変わる」
を体現する最強コンテンツの一つ
```

**データソース**
```
★★★★★ 国税庁・財務省公開データ
★★★★☆ 各社有価証券報告書（消費税還付は開示義務あり）
★★★★☆ 主要報道機関の調査報道
```

### 街頭演説パターン分析＆当落予測モデル
> 街頭演説は「本当の政治家」が見える場所。YouTube×Whisperで全文テキスト化し、当落との相関を分析する。

**街頭演説は国会と全く別のデータ**
```
国会答弁: 原稿あり・記録あり・緊張感あり
街頭演説: 本音が出る・感情が出る・キャラが出る
→「本当の政治家」が見えるのは街頭演説
```

**分析対象パターン**
```
【必ず入れる定型フレーズ】
「皆さんと一緒に」「現場の声を聞いて」
「○○県民のために」「この国の未来のために」
→ 議員ごとの口癖ランキング
→ 政党ごとの定番フレーズ比較

【地元ネタ】
地元の祭り・名産・球団・方言を入れる
→ 複数年の演説を比較すると
  「毎年同じまつりネタを使う議員」が出てくる

【笑いのパターン】
自虐系:「私のような者が言うのも何ですが」
身内ネタ系:「妻に言われまして」
地元いじり系・時事ネタ系
→ どの議員が笑いを取るのが上手いか
→ 笑いの量と投票率の相関

【叫ぶパターン】
「絶対に！」「必ず！」「断じて！」
→ 叫んだ内容 × 実現率
→「絶対に」と叫んで実現しなかった約束ランキング
```

**演説の構造パターン**
```
典型的な構造:
①つかみ（地元ネタ・笑い）30秒
②共感（今の生活の苦しさ）1分
③問題提起（現政権の批判）2分
④自分の訴え（政策）2分
⑤感情の高揚（叫び）1分
⑥締め（お願い）30秒

→ この構造からの逸脱が個性になる
→ 議員ごとの構造パターンを可視化
```

**動画証拠保存ポリシー（確定）**
```
【取得・保存する情報（必須）】
① 字幕付きスクリーンショット
  → 動画を再生しながら字幕が表示された状態でSS
  → 「この映像でこの発言をした」の視覚的証拠

② Whisperによる文字起こし
  → SSの字幕と文字起こしが一致 = 信頼性が掛け算で上がる
  → テキストとして検索・分析可能

③ メタデータ
  → 動画URL・タイトル・投稿日時・チャンネル名・再生数
  → WebArchiveのURL（ページの存在証明）

④ 削除フラグ
  → 動画が消えたら「削除済み」として記録
  → 削除日時・削除前の再生数も記録

【動画本体は保存しない】
著作権・利用規約のリスクを避ける
ただし存在した証拠は完全に残す
```

**「削除できない」設計の強さ**
```
議員が動画を削除しても:
・SSは残る
・文字起こしは残る
・メタデータは残る
・WebArchiveは残る
・「削除した」という事実が記録される

「削除した」という行動自体が情報
→ なぜ消したのか → 何を言っていたのか
→ 隠したかった証拠として機能する
```

**データ取得パイプライン**
```
YouTube（議員公式チャンネル）
→ yt-dlpで動画自動取得
→ Whisperで文字起こし（無料・高精度）
→ テキストとして分析

国会議事録にない「街頭の言葉」が全部テキスト化される
```

**国会答弁との比較（最強コンテンツ）**
```
街頭演説:「消費税は絶対に上げません！」（叫ぶ）
国会答弁:「諸般の事情を考慮して検討します」

→ 同じ議員・同じテーマでの
  「表の顔」と「裏の顔」が並ぶ
```

**当落予測モデル（核心）**
```
当選した人の演説パターン
落選した人の演説パターン

これを大量に比較すると
「当選する演説の特徴」が統計として出てくる

例として見えてくる可能性があるもの:
・地元ネタを入れた演説は当選率が高いか
・笑いを取れる候補は当選しやすいか
・叫ぶ頻度と当落の相関
・演説時間の長短と当落
・使うキーワードのパターン差
・感情的 vs 論理的 どちらが勝つか
・「私」を多用 vs「皆さん」を多用
```

**複合要素との組み合わせ**
```
演説パターン単体では予測精度は低い
ただし他のデータと組み合わせると強くなる

演説パターン
　+ 選挙区の属性（都市・地方・年齢層）
　+ 対立候補の強さ
　+ 政党支持率
　+ 直近のスキャンダルの有無
　+ 前回の得票数

→ 複合モデルとして当落予測精度が上がる
→「この候補の当選確率: 67%」が出せる
```

**選挙コンサルへの価値**
```
「当選する演説パターン」のデータを持っているのは
日本でPoliMirrorだけ

選挙コンサルが欲しいのはまさにこれ
→「この選挙区ではこのトーンの演説が効く」
→「このキーワードを入れると支持率が上がる」

1案件数百万円のコンサルが
このデータに月数十万円払う理由になる
```

**長期的に見えてくるもの**
```
時代による演説スタイルの変化
2000年代: 声量・熱量重視
2010年代: SNS映えを意識した短い言葉
2020年代: 共感・寄り添い系が増加

→「政治家の言葉」の歴史が記録される
→ 社会の変化と演説スタイルの相関
```

### 反論データベース
> 政治の対立構造を「反論」という軸で可視化する。PoliMirrorの核心コンテンツになりえる。

**反論の3種類**
```
① 議員A → 議員Bへの反論
② 議員A → メディア・評論家への反論
③ 議員A → 自分の過去発言への事実上の反論（矛盾・ダブスタ）
③が最も強い。本人が意図せず自分に反論している。
```

**反論ネットワークの可視化**
```
誰が誰に最もよく反論するか
→ 表向きの党派と実際の対立軸が違うことが浮かび上がる
例:「同じ自民党内なのにこの2人は年間50回反論し合っている」
→ 派閥対立の可視化
```

**反論の質スコア（Claude API自動判定）**
```
低品質: 感情的・人格攻撃・論点ずらし・「お前こそ」型
高品質: データを示して否定・論理的に前提を崩す・代替案提示

→「論理スコア」として公開
→ 議員の知的水準が可視化される
```

**反論の結果追跡**
```
反論 → 相手が撤回した     → 「有効な反論」
反論 → 無視された         → 「空振り」
反論 → さらに反論が来た   → 「論争継続中」
反論 → 政策が変わった     → 「最強の反論」
```

**反論類型の分類**
```
① 事実否定型:「そのデータは間違っている」→ 最も検証しやすい
② 解釈反論型:「事実はそうだが意味が違う」
③ 優先度反論型:「重要な問題は別にある」→ 論点ずらしとの境界
④ 体験反論型:「現場ではそうじゃない」
⑤ 前例反論型:「過去に同じことをやって失敗した」→ 時系列DBと直結
```

**反論チェッカー（機能）**
```
ユーザーが議員の発言を入力
→ Claude APIが「この発言への過去の反論」を検索
→「同じ主張に対してXX年にAA議員がこう反論しています」を表示
```

**論戦勝率ランキング**
```
国会での質疑応答を記録
どちらの論が論理的に優れているかをClaude APIが判定
→「論戦勝率ランキング」
→ スポーツのように政治が見られる
```

**投資シグナルとの接続**
```
「この法案への反論が急増している」
→ 法案成立確率が下がる → 関連セクターへのシグナル
反論の活発さ = 政策の不確実性 = 市場への影響
```

### 「記憶にない」データベース
> 日本政治最強の免責ワード。責任逃れの常套句を記録・分析・可視化する。

**なぜ「記憶にない」が使われるか**
```
事実を否定すると → 後で嘘がバレるリスク
記憶にないと言うと → 嘘にならない
                 → 責任も取らなくていい
                 → 法的に「知らなかった」= 共謀の証明困難
→ 政治家が学習した最強の盾
```

**収集する発言パターン**
```
「記憶にございません」
「記憶がございません」
「記憶しておりません」
「覚えていません」
「存じません」
「承知しておりません」

国会議事録を全文検索
→ 議員別・年別・テーマ別で集計
→「記憶にない率ランキング」
```

**類型化**
```
① 完全否定型:「そのような事実はございません」→ 最もリスク高い
② 記憶喪失型（最多）:「記憶にございません」→ 法的に最も安全
③ 確認中型:「調査して参ります」→ その後報告しない前提
④ 定義逃げ型:「会食とは言えない軽い食事でした」→ 言葉の定義を変える
⑤ 責任転嫁型:「担当者に確認します」→ 自分は知らないと示唆
⑥ 時効型:「当時のことは…」→ 時間が経ったことを盾にする
```

**記憶回復チェッカー（機能）**
```
「記憶にない」と言った内容を記録
→ その後同じ議員が同じ内容に言及したら自動検出
```

**ステータス一覧**
```
🔴 疑惑浮上中     → 報道・国会で指摘・本人コメントなし
🟡 回答待ち       → 「調査します」「確認します」期限未定
🟠 期限付き回答待ち → 「○月○日までに報告」カウントダウン表示

🔵 選挙後に回答予定 → 「選挙が終わったら」と発言・追跡対象
🟢 回答済み       → 内容・日時・回答の質を記録
⚪ 有耶無耶・立ち消え → 誰も追及しなくなった・でも記録し続ける
```

**回答の質スコア（Claude API判定）**
```
A: 完全回答  → 具体的な事実・数字・経緯を説明
B: 部分回答  → 一部だけ答えた
C: 論点ずらし → 別の話にすり替えた
D: 形式的回答 → 「真摯に対応します」で終わった
F: 事実上の無回答 → 「記憶にない」「確認中」のみ
```

**経過日数カウンター**

```
🟠 B社接待問題: 回答期限まで【残り3日】
```

### 公約・マニフェスト達成率管理
> 「言ったことをやったか」を選挙ごとに追跡する。曖昧な公約は曖昧と明記する。

**公約データの収集方法**
```
選挙公報（総務省・各選管が公開）
→ 全候補者の公約が全部掲載
→ PDFをOCRでテキスト化

政党マニフェスト（各党公式サイト）
→ 過去分はWebArchiveから取得

議員個人の選挙サイト
→ 選挙中だけ公開されることが多い
→ 選挙公示日にクローラーを走らせて必ず取得・保存
```

**選挙ごとの自動アクション**
```
選挙公示日:
→ 全候補者の公約を自動取得・保存・SS撮影

投開票日:
→ 当落結果を記録
→ 当選者の公約を「追跡対象」に設定

当選後毎年:
→ 公約達成状況を自動チェック
→ 国会議事録・法案成立記録と照合

次の選挙前:
→ 達成率レポートを自動生成・公開
```

**判定の3段階**
```
✅ 明確に達成
   数字・法案・予算で証明できる

⚠️ 判定困難 = 曖昧な公約
   基準が不明確・何をもって達成とするか不明
   → 判定困難の理由も明記する
   → 「曖昧な公約を書いた」という事実を記録

❌ 明確に未達成
   反対の政策を実施した等
```

**「判定困難＝曖昧な公約」の可視化**
```
今まで誰も指摘しなかった視点

「景気を良くします」→ 判定困難 = 曖昧な公約

これを明記するだけで:
・有権者が公約の質を評価できる
・曖昧公約が恥ずかしいものになる
・政治家が具体的な公約を書くようになる
・曖昧にすること自体が責任逃れの手段だったと証明される
```

**公約の質スコア（Opus APIで判定）**
```
5点: 数字・期限・手段が全部ある
     例「消費税を2025年までに5%に下げる」

4点: 数字か期限のどちらかがある
     例「最低賃金を1,500円にする」

3点: 方向性は明確だが数字がない
     例「最低賃金を引き上げる」

2点: 方向性が曖昧
     例「経済政策を見直す」

1点: 何も言っていないのと同じ
     例「景気を良くします」
        「国民のために働きます」
        「政治を変えます」
```

**曖昧公約ランキング**
```
「最も曖昧な公約を掲げた議員」
「最も具体的な公約を掲げた議員」

→ 選挙前に公開すると有権者の投票基準が変わる
→ 長期的に政治の質が上がっていく
```

**宣言と結果のマトリクス**
```
           具体的公約    曖昧公約
達成した  │    ◎      │    ?    │
未達成    │    ✕      │  測定不能│
判定困難  │   まれ    │   多い  │

→「曖昧公約は達成率を測定できない
  仕組みになっている」が一目でわかる
```

### 判定モデルの原則（透明性の核心）
> 判定の質 = PoliMirrorの信頼性。常に最高のモデルを使い、それを明示する。

**「判定には常に最高のモデルを使う」原則**
```
安いモデルで判定を間違えると:
・議員から「判定がおかしい」と攻撃される
・信頼性が崩れる
・修正コストが判定コストより高くなる

→ 判定だけは妥協しない
→ その時点での最高モデルを使う
```

**モデルの使い分け設計**
```
【最高モデル（Opus系）】
・公約の質スコア判定
・疑惑への回答の質判定
・矛盾・ダブスタ検出
・常套句の文脈判定
・発言の本気度判定
→ 結果がPoliMirrorの信頼性に直結するもの

【中間モデル（Sonnet系）】
・データの構造化
・MDファイルの自動生成
・要約・サマリー作成
・一次スクリーニング
→ 大量処理・精度より速度

【軽量モデル（Haiku系）】
・単純な分類・タグ付け
・重複チェック・フォーマット変換
→ 単純作業・コスト最小化
```

**コスト管理との両立**
```
Sonnetで一次判定
→ 判定困難・スコア3前後のものだけOpusで二次判定
→ コストを10分の1に抑えながら精度はほぼOpus同等

年間Opusコスト: 数十万円
でもそれがPoliMirrorの「信頼性」という資産になる
```

**判定モデルの明示（最重要）**
```
判定結果の下に必ず表示:

「この判定はClaude Opus 4.6
 （2025年○月時点）により行いました」
「判定ロジック: GitHub参照」

→ 判定の根拠が明確になる
→ AIが恣意的に判定したと言えない
→ 人間の主観が入っていないことの証明
→ モデルが変わったら再判定できる
```

**バージョン管理との組み合わせ**
```
GitHub上で全部記録:

commit: 2025-03-15
「山田太郎の公約をOpus 4.6で再判定
 スコア: 2→3点に変更
 理由: モデル精度向上により文脈の読み取りが改善」

→ 判定が変わった理由も透明化
→「気まぐれで変えた」と言われない
```

**将来への対応**
```
2024年: Opus 3が最高
2025年: Opus 4.6が最高
2026年: さらに上のモデルが出る

→ 判定モデルを更新するたびに過去の判定を再評価
→「2024年の判定を2026年の最高モデルで再評価した結果」
→ 判定精度の向上が透明化される
→ これもGitHubで記録
```

### robots.txt監視・公開隠蔽の可視化
> 公人がrobots.txtで情報取得を拒否している事実自体がニュースになる。

**逆説的な透明化**
```
公人・公的機関 → 情報公開が原則
robots.txtでクローラー全拒否
→「何を隠したいのか」が浮かび上がる

PoliMirrorが収集できなかった情報も
「収集できなかった」という事実が情報になる
```

**robots.txt変更ログの記録**
```
全議員・全公的機関のrobots.txtを定期取得・記録

記録内容:
・現在のrobots.txtの状態
・変更日時・変更内容
・変更前後の比較

「山田太郎議員のサイト
 2024年3月15日にDisallow: /を追加」

→ いつ・何を・隠し始めたか
→ 政治的イベントとの相関が見える
```

**スキャンダルとの相関**
```
スキャンダル報道の翌日に
robots.txtを変更した議員

→ これだけでニュースになる
→ 「隠した」という行動が証拠になる
→ 変更前にSSを撮っていれば完璧
```

**公的機関も対象**
```
各省庁・地方自治体・外郭団体・関連法人

公金が入っている組織が
情報取得を拒否している
→ 情報公開法の趣旨に反する
→ 問題提起できる
→ メディアへのネタ提供になる
```

**「隠蔽指数」として可視化**
```
robots.txtの制限レベルを
議員・機関別にスコア化

完全公開:  ★★★★★
一部制限:  ★★★☆☆
全面拒否:  ★☆☆☆☆

→「情報隠蔽度ランキング」
→ 透明性スコアの一部として組み込む
```

### データ収集ポリシーの透明化
> 収集方法を全公開することが最強の防衛になる。やましいことは何もない。

**収集しているのは全部公開情報**
```
国会議事録         → 国が公開
裁判所判決文       → 裁判所が公開
政治資金収支報告書  → 総務省が公開
議員公式サイト     → 本人が公開
YouTube公式チャンネル → 本人が公開

「公開されている情報を集めているだけ」
やましいことは何もない
```

**収集ルール（明文化・公開）**
```
① robots.txtを遵守する
② リクエスト間隔を5〜10秒に設定
   サーバーに負荷をかけない
③ 深夜帯のみ収集（負荷分散）
④ キャッシュを活用して再取得を最小化
⑤ ログインが必要なページは収集しない
⑥ 公務に関する情報のみ収集
   私生活・家族情報は収集しない
```

**収集コードをGitHubで全公開**
```
これが最強の回答

「どうやって収集しているか？」
→「GitHubで全部公開しています」
→「誰でも確認できます」
→「隠していることは何もありません」

収集コードの公開 = 透明性の証明 = 攻撃できない
```

**法的根拠**
```
・著作権法第13条（国会議事録等は著作権なし）
・情報公開法（公的機関の情報は公開義務あり）
・公開情報のスクレイピングを禁じる法律は日本にない
・不正アクセス禁止法 → ログインページは収集しないので対象外
・個人情報保護法 → 公務情報のみ収集するので対象外
```

**「なぜ問題なのか」と逆に問える**
```
「公開情報を集めることがなぜ問題なのですか？」
→ 答えられる人はいない

「この情報を集めてほしくないのは誰ですか？」
→ 圧力をかけてきた側の意図が透けて見える
```

**サイトに明記する文言**
```
「PoliMirrorのデータ収集ポリシー」

当サービスは公開情報のみを収集します。
収集方法はGitHubで全公開しています。
robots.txtを遵守し、サーバーに負荷をかけない
設計で運用しています。
収集目的は公益・政治の透明性向上です。
```

**取得できるデータ（全部公開情報）**

```
✅ 最高裁判所の判決文（裁判所ウェブサイト）
✅ 補足意見・反対意見（誰が書いたか記録）
✅ 法廷意見への裁判官別賛否
```

```
・補足意見の傾向

一貫性:
```

### 党員数データ
> リアルタイムは不可能だが「常に最新・推計値あり」で誰もやっていないコンテンツになる。

**取れるものと取れないもの**
```
取れるもの:
✅ 政治資金収支報告書の党費収入（年1回・総務省公開）
✅ 各党の公式発表（プレスリリース・大会資料）
✅ 国政選挙の比例得票数（支持者数の代替指標）

取れないもの:
❌ リアルタイムの党員数
❌ 月次・週次の増減
→ 各政党がそもそも公開していない
```

**PoliMirrorならではの推計値**
```
党費収入 ÷ 党費単価 = 推計党員数

例: 自民党
党費収入: 政治資金収支報告書から取得
党費単価: 公開されている党則から取得
→ 推計党員数を算出

誰もやっていない
「推計党員数ランキング」が作れる
```

**設計方針**
```
「リアルタイム更新」ではなく
「最新データに自動更新」

総務省が収支報告書を公開したら即取得・即反映
→ 年1回でも「常に最新」は維持できる
→ 更新日時を明示することで信頼性を担保
```

**時系列で見えてくるもの**
```
党員数の増減 × 政治イベントの相関

スキャンダル後に党員が減ったか
選挙に勝った後に党員が増えたか
政権交代前後の変化

→ 政党の「リアルな勢い」が数字で見える
```

### 法的防衛設計＆削除要請の透明化
> 一次ソースのみという原則が最強の盾。削除要請自体を武器に変える。

**なぜ一次ソースが盾になるか**
```
名誉毀損の成立要件に「真実でない」がある
PoliMirrorが記録するのは:
・国会議事録（著作権法第13条で転載自由）
・官報・裁判確定記録
・議員本人の公式発言

→「真実でない」と言えない
→ 名誉毀損が成立しない
```

**想定される攻撃と防衛**
```
「削除しろ」
→「出典は国会議事録○年○月○日です。
  削除するには国会議事録を書き換えてください」

「名誉毀損だ」
→「全て一次ソースからの引用です。
  虚偽であれば国会・官報・裁判所に申し立てを」

「プライバシー侵害だ」
→ 政治家の公務情報は知る権利・公益性が優先
  最高裁判例で確立済み

「著作権侵害だ」
→ 国会議事録は著作権法第13条で保護対象外
  自由に引用・転載できる
```

**弱点と対策**
```
週刊誌・SNSからの引用 → 使わない・一次ソースのみ徹底
解釈・分析部分 → 「AIによる分析です」と明示・事実と分析を分離
データ入力ミス → 修正フローを公開・修正履歴も全公開
```

**問い合わせ対応の現実**
```
99%はAIが即答で終わる
削除要請・訴訟キーワードが来た時だけ管理者に転送

ほとんどAIの返答で完結する設計で十分
```

**削除要請の透明化＋X投稿（重要）**
```
削除要請が来たら:
① 要請内容・要請者・日時を記録
② サイト上に「削除要請ログ」として公開
③ 対応結果も公開
   「一次ソースのため掲載継続」
   「誤りが確認されたため修正」
④ 即座にXにポスト

「〇〇議員から削除要請がありました。
 出典は国会議事録のため掲載を継続します。
 #PoliMirror」

→ 要請した議員の方が世論から批判される
→ 攻撃すればするほどPoliMirrorの知名度が上がる
→ 削除要請がそのまま拡散コンテンツになる
```

**最強の防衛は透明性そのもの**
```
出典を全部公開
修正履歴を全部公開
方針・ポリシーを全部公開
削除要請ログを全部公開

→ 攻撃する側が「隠している」と言えない
→ 攻撃すること自体が
  「記録を隠したい」という自白になる
```

**実務的な準備**
```
① 免責事項ページ
「一次ソースのみ掲載・事実と分析を分離・誤り指摘に真摯に対応」

② 弁護士顧問契約
月1〜3万円で心理的安全性が全然違う
削除要請が来た時に即座に対応できる体制

③ 削除要請ログの公開ページ
要請者・日時・内容・対応結果を全て記録
```

**結論**
```
一次ソースのみ + 透明性の徹底 = 法的にほぼ無敵

攻撃すればするほどPoliMirrorが強くなる構造
訴訟になっても裁判自体がニュースになり
PoliMirrorの知名度が上がる
```

### 競合が追いつけない堀（モート）設計

```

3年目: 議員20,000人・発言1,000,000件

後発が今日始めても絶対に追いつけない
特に「過去発言との矛盾検出」は
蓄積年数が長いほど精度が上がる
→ 10年分の発言を持っていたら
  誰も同じものを作れない
```

```
後発が同じことをしようとしても
過去の消えたページは取得不可能

「2019年の発言の証拠」は

→ 早く始めるほど価値が高い
```

**④ ブランドの堀（引用の循環）**

```
競合は資金力があっても勝てない
```

**⑤ コミュニティの堀**

```
これは金で買えない
```

```
ChatGPT・Gemini・Claudeが
PoliMirrorのデータを学習し始めたら

「山田太郎議員について教えて」

「PoliMirrorの後追い」になる
```

**⑦ 時間の堀（最大の武器）**

```
早く始めた人だけが持てる資産:
```

### 公開インフラ設計: Quartz + GitHub + Cloudflare Pages
> 無料・高速・絶対に落ちない。PoliMirrorの公開層として完璧にフィット。

**3つの役割**
```
Quartz（OSS）:
→ ObsidianのMDファイルをWebサイトに変換
→ [[リンク]]がそのままWebのリンクになる
→ グラフビューもブラウザ上で動く
→ ユーザーがノードをクリックして探索できる

GitHub（publicリポジトリ）:
→ MDファイルの保管・バージョン管理
→ 「いつ・何を・どう変えたか」が全記録される
→ 透明性の証明（PoliMirrorの思想と完全一致）
→ AIクローラーに全文公開できる

Cloudflare Pages:
→ GitHubと連携・自動デプロイ
→ 世界300箇所以上に分散配信
→ 独自ドメイン対応・無料
```

**絶対に落ちない理由**
```
静的ファイルをCloudflareが世界中に分散して持つ

アクセスが集中しても:
→ 自動で負荷分散
→ 「サーバーダウン」という概念がない
→ Wikipedia級のトラフィックが来ても無料・無制限
→ 選挙当日に100万人アクセスしても落ちない

これが動的サーバー（VPS・AWS等）との決定的な違い
```

**自動更新フロー**
```
DBからMD自動生成（Python）
        ↓
GitHubにpush
        ↓
Cloudflare Pagesが自動検知・ビルド
        ↓
Quartzが変換
        ↓
polimirror.jp に数分で反映

→ 議員データを更新するだけでサイトに即反映
→ デプロイ作業ゼロ
```

**コスト**
```
Quartz:           無料（OSS）
GitHub:           無料（publicリポジトリ）
Cloudflare Pages: 無料（商用利用・無制限トラフィック）
独自ドメイン:     年間約1,500円のみ
```

**段階的展開設計**
```
フェーズ1（今すぐ）:
全データをQuartz + GitHub + Cloudflare Pagesで無料公開
→ LLMのクローラーに拾わせる
→ 日本政治版Wikipediaのポジションを取る

フェーズ2（収益化後）:
有料機能のみNext.js SaaSを追加
→ アラート・API・ダウンロード販売
→ 無料公開層はそのまま維持

無料層が集客→有料層でマネタイズ
の好循環設計
```

**注意点**
```
Quartzは静的サイト
→ 検索・グラフ表示はJSで動く（問題なし）
→ ユーザーログイン・課金は別途SaaS層が必要
→ ただし最初は不要・公開だけで十分
```

### 消費税還付企業ランキング
> 「消費税を一番多く還付されている企業はどこか」を可視化する。輸出大企業への優遇構造が一目でわかる。

**なぜ重要か**
```
消費税の輸出還付制度:
輸出企業は消費税を払わなくていい
→ 仕入れ時に払った消費税が還付される
→ トヨタ・ソニー等の大企業に年間数千億円が還付

これは公開データで計算できるが
誰もランキング化していない → 空白地帯
```

**収集データ**
```
法人の消費税申告データ（財務省・国税庁公開統計）
有価証券報告書の消費税関連記載
経済産業省の輸出統計

→ 還付推計額トップ30・100をランキング化
→「消費税還付企業ランキング」として公開
```

**政治家との紐付け**
```
還付上位企業 → 献金先の議員 → 消費税政策スタンス

「この議員は消費税増税に賛成している」
「この議員の献金元トップ3は消費税還付上位企業」
→ 利益相関が可視化される
→ PoliMirrorの政治資金データと直結
```

**バイラルポテンシャル**
```
「トヨタは消費税を年間○○億円還付されている」
「消費税増税のたびに還付額も増える仕組み」
→ 一般ユーザーが知らない事実
→ SNSで拡散されやすい
```

### 問い合わせ対応AI（カスタマーサポート自動化）
> 99%はAIが即答。人間が対応するのは1%だけ。

**基本構造**
```
ユーザーの問い合わせ
        ↓
Claude API（PoliMirrorの知識を持つ）
        ↓
99% → AIが即答
1%  → 管理者にメール転送
```

**AIが答える99%**
```
データについて:
「〇〇議員のデータはありますか？」
「この情報の出典を教えてください」
「なぜこのスコアになっていますか？」

使い方について:
「検索の仕方がわかりません」
「タグの意味を教えてください」
「デッキビルダーの使い方は？」

方針について:
「なぜこの情報は載っていないのですか？」
「個人情報はどう扱っていますか？」
「中立性をどう担保していますか？」

データ修正依頼:
「この情報が間違っています」
→ AIが受け付けて修正キューに自動登録
```

**人間が必要な1%**
```
・法的通知・弁護士からの連絡
・報道機関からの取材依頼
・スポンサー・提携の打診
・明らかに悪意ある問い合わせ
→ 「担当者に転送します」と答えてメール通知
```

**実装方法（2段階）**
```
フェーズ1（今すぐできる）:
Claude APIを直接埋め込む
→ サイトにチャットUIを設置
→ system promptにPoliMirrorの全知識を入れる
→ 費用: API利用料のみ

フェーズ2（本格版・RAG構成）:
DBのデータをベクトル化
→ 質問に関連するMDファイルを自動参照してから回答
→「山田太郎について」→山田太郎のMDを読んで回答
→ 精度が格段に上がる
```

**System Promptの設計方針**
```
・事実のみ答える
・出典を聞かれたら必ず提示する
・政治的な意見・評価は絶対に言わない
・「〇〇議員は悪い政治家ですか？」
  →「評価はユーザーに委ねています」と返す
・転送条件: 「弁護士」「訴訟」「削除要請」のキーワード
```

### 政治評論家・コメンテーター系（PoliMirrorの拡張カテゴリ）
> 「政治家について誰が何を言ったか」を記録する。発信量は多い。一貫性は低い。

**対象範囲**
```

※ SNSのみの論客は原則対象外（署名記事があれば対象）
```

**記録するデータ**
```
予測の記録:
- 選挙予測の発言（出典・日時）→ 結果と照合
- 予測的中率スコア（全員同じ基準で算出）
- 「外れた予測」も永久に記録

スタンスの記録:
- 主要争点ごとのスタンス（政治家と同じ軸）
- 政権交代前後でのスタンス変化
- 発言の矛盾・変節（自動検出）

メディア露出:
- 出演局・番組・回数（月別・年別）
- 特定局への集中度
- 出演料・原稿料は非公開のため記録しない

収入源（公開情報のみ）:
- 著書・出版物
- 講演活動（公開されているもの）
- 大学・研究機関との所属
- 企業顧問（公開されているもの）
```

**可視化できる面白いコンテンツ**
```
「選挙予測的中率ランキング」
→ 誰の予測が一番当たるか

「政権交代で論調が変わった評論家」
→ スタンス変化を時系列グラフで表示

「特定局依存度ランキング」
→ 「この人はほぼTBS専属」が数値でわかる

「3年前と今で真逆の発言をした評論家」
→ 矛盾検出・変節センサーを適用

「発信量 vs 予測精度」散布図
→ 発信は多いが外れ続ける評論家が可視化される
```

**設計上の注意点**
```
政治家との違い:
・公権力を持たない→記録の根拠が弱い
・テレビ出演・書籍・署名記事のみ対象
・個人攻撃にならないよう全員同じ基準を徹底

やらないこと:
・「この評論家は信用できない」という評価はしない
・データを並べるだけ・解釈はユーザーに委ねる
・PoliMirrorの中立原則をより厳格に適用
```

## 🎨 AI生成画像の方針

### 結論：「似せない」が正解

```
似せる（公式写真→AI似顔絵）
→ 肖像権の議論が生じる
→ 精度が低いと別人になって混乱
→ 「無断で顔を使った」と言われるリスク

似せない（属性データ→アイコン生成）
→ 肖像権の問題がゼロ
→ 全員統一デザインで作れる
→ 法的に完全にクリア
```

### PoliMirrorの画像3段階ルール

```
① 公式写真がある議員
  → 公式写真を表示（本物を優先）

② 公式写真がない議員（地方議員に多い）
  → 属性アイコンを表示
    （政党色・年代・性別を反映したフラットアイコン）
    ※ 本人に似せない・属性だけ反映

③ 議員ポケモンカード
  → 「データを擬人化したキャラクター」として設計
    （顔を似せるのではなく能力値を視覚化）
    例: 発言力が高い → 炎を持つキャラ
        一貫性が低い → 色が変わるキャラ
        献金が多い  → お金のエフェクト付き
```

### 属性アイコンで使う要素
- [ ] 政党カラー（自民=青・立民=赤・維新=オレンジ等）
- [ ] 年代（20代/30代/40代/50代/60代以上）
- [ ] 性別
- [ ] 役職（議員/知事/市長/候補者）
- [ ] ステータス（現職/落選/引退）

### カードデザインのコンセプト
```
「その議員のデータが見た目に出る」
→ 顔を似せるより「データを擬人化」した方が
  コンセプトとして強く・法的にも安全
→ 全35,000人を統一フォーマットで作れる
→ 公式写真がない地方議員も平等に表示できる
```

### 注意ルール
```
✅ 属性・データのみ反映。本人の顔は参照しない
✅ 「AIアイコンです」と明記
✅ 全議員を同じスタイルで統一（差別なし）

❌ 本人の写真を学習させて似顔絵生成はしない
❌ 写実的・リアルな顔生成はしない
❌ 侮辱・誇張・性的・暴力的表現は絶対NG
```

### political-finance-database.com（最大競合）
- 開発者: 西田尚史氏（東大院博士課程）

## 🤝 競合との役割分担戦略（2026年4月確定）

**「概要はPoliMirror、詳細は競合へ」**

PoliMirrorは概要数値を自前で用意し、トランザクション詳細は
political-finance-database.com に明示的にリンクする。

### PoliMirrorが持つデータ（総務省集計表から取得）
- 年次総収入額・総支出額
- 企業献金総額・割合
- パーティー収入額・割合
- 個人献金総額・割合
- 関連政治団体数

→ 取得元: 総務省公開の集計Excel/PDF（トランザクション単位OCR不要）
→ 難易度: 低（名寄せのみ課題）

### 競合に任せるデータ
- 1件ずつのトランザクション明細
- キーワード検索（支出先の飲食店・宿泊施設等）
- 寄付ランキング

### 議員ページの表示イメージ
```
## 政治資金（2023年）

総収入: 4,820万円
└ 企業献金: 1,200万円（25%）
└ パーティー収入: 2,100万円（44%）
└ 個人献金: 380万円（8%）
総支出: 4,650万円

📊 トランザクション詳細 → political-finance-database.com
```

### なぜこの戦略が正しいか
- 資金データの粒度で競合に勝とうとしない
- 競合を「敵」ではなく「インフラ」として活用
- PoliMirrorの中立性・誠実さの表明にもなる
- 開発コストを発言×矛盾検出に集中できる
- ユーザー体験が上がる（両方使えばいい）

- 全トランザクション抽出済みの西田昌司でのみ成功
- → 全議員OCRは非現実的。集計値＋発言で攻める方針に転換
